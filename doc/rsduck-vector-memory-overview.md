# RSDuck：把 Agent 记忆从“能存”变成“能召回”

语言：[English](rsduck-vector-memory-overview.en.md) | 中文

大模型 Agent 真正难的部分，往往不在一次对话，而在如何从大量历史记忆中找回当前最有价值的少量内容。RSDuck 在 DuckDB 之上提供固定维度向量、受管 HNSW 索引和专用 Vector API，用于承担这一段“语义召回”工作。

它不是要替代 MySQL 等业务数据库，而是作为记忆检索加速层：业务库保存正文、状态、权限和版本；RSDuck 根据语义找出有价值的 `memory_id`，业务服务再按 ID 回事实源读取正文。这种分工既避免了两处保存两份事实，也让 Agent 可以在大量记忆中快速找到相关上下文。

## 1. Agent 为什么需要向量记忆

传统数据库查询擅长“条件相同”：例如按用户、时间、标签或关键词过滤。但用户的表达常常不是精确匹配。

例如，历史记忆中记录了“客户希望将月度经营报表改为周报，并在周一上午发送”。用户今天问：“上次那个报告要什么时候给对方？”关键词未必相同，但两句话表达的是同一件事。

向量记忆解决的就是这种**语义相近而字面不同**的问题：

1. Embedding 模型将文本转换为一串浮点数，即向量。
2. 语义接近的文本，其向量在高维空间中通常也更接近。
3. 用当前问题的向量检索历史向量，即可找到语义相关的记忆。
4. RSDuck 返回 `memory_id + distance`，Agent 再从业务库读取原文、权限和最新业务状态。

这使 Agent 的回答不只依赖当前窗口中的聊天记录，还能稳定利用经过沉淀的长期记忆。

## 2. RSDuck 在整体架构中的位置

```text
职业 Agent / 业务服务
        |
        | 写入业务事实、状态、正文
        v
MySQL 等关系型业务库  <-------------------+
        |                                 |
        | Outbox 事件                      | 按 memory_id 批量读取正文
        v                                 |
Embedding Worker -- 生成固定维度向量 --> RSDuck
                                             |
                              HNSW 语义检索 | 返回 memory_id + distance
                                             v
                                      Agent 组装上下文
```

这里有两个关键边界：

- **MySQL 是事实源。** 记忆正文、删除状态、可见范围、租户关系和业务版本以业务库为准。
- **RSDuck 是可重建的检索层。** 它保存向量、索引定义和运行状态；即使索引需要重建，也不应丢失业务事实。

因此，RSDuck 返回的是候选记忆，而不是绕过业务权限直接返回正文。业务服务在回读正文时仍需再次校验租户、Agent 和可见性。

## 3. 从文本到相似度：向量检索的原理

### 3.1 Embedding：把语义映射到坐标

Embedding 模型会把一段文本转换为固定长度的数值数组，例如 1,536 维：

```text
“客户要求周一发送周报”
  -> [0.12, 0.35, -0.08, ..., 0.78]
```

向量中的每个数本身通常没有业务可读含义；它们组合起来表示文本在语义空间中的位置。使用同一个模型处理的“周报什么时候发”和“周一上午给客户发经营报告”会落在相近的位置。

RSDuck 使用 `FLOAT[N]` 表示这种契约：`N` 是确定的维度，例如 `FLOAT[1536]`。维度不是可随意变化的字段属性，而是向量空间的一部分。一个向量空间只能对应一个模型、模型版本、维度和距离算法；模型升级时要创建新空间、重新生成向量，再切换流量。

### 3.2 距离：如何定义“相近”

检索会计算查询向量与历史向量的距离。以余弦距离为例，它关注两个向量的方向是否一致，通常适合文本 Embedding。

```sql
SELECT
    memory_id,
    list_cosine_distance(
        embedding,
        [0.12, 0.35, 0.78]::FLOAT[]
    ) AS distance
FROM agent_memory_vector
WHERE tenant_id = 1001
  AND agent_id = 2001
ORDER BY distance
LIMIT 20;
```

这是一条**精确检索**：在租户和 Agent 的范围内逐行计算距离，结果是精确的，但数据量很大时成本会随向量数线性增长。它很适合调试、验收和基准测试。

线上召回通常使用 ANN（Approximate Nearest Neighbor，近似最近邻）索引。ANN 用极小概率错过部分边缘候选，换取数量级更好的延迟和更可控的资源使用。

## 4. HNSW：为什么它能更快

HNSW 的全称是 Hierarchical Navigable Small World，直观理解为“分层的近邻导航图”。

它不是把所有向量逐个扫一遍，而是预先将相近向量连成图：

```text
上层：少量远距离导航节点     A -------- B -------- C
                                  \                 /
下层：更多局部近邻节点       A1--A2--A3 ... B1--B2 ... C1--C2
```

一次检索大致经历：

1. 从高层少量节点开始，快速找到离查询向量较近的区域。
2. 逐层向下进入更密集的局部近邻图。
3. 在最后一层扩展足够多的候选，返回距离最小的 `top_k` 个记忆。

这很像先用城市地图定位到一个城区，再进入街道寻找具体地址，而不是从全国每一户逐一询问。

HNSW 的常用参数包括：

- `m` / `m0`：每个节点保留的近邻数量。越大通常召回越好，但索引更占内存、构建更慢。
- `ef_construction`：建索引时的候选搜索宽度。越大通常图质量越好，但构建更慢。
- `ef_search`：查询时的候选搜索宽度。越大通常召回越好，但单次查询延迟更高。

不存在放之四海而皆准的参数。正确做法是用真实维度、真实数据规模和业务 `top_k`，同时比较 Recall、p95 延迟和 RSS 内存占用。

## 5. RSDuck 与纯 DuckDB 向量使用的区别

RSDuck 并不是另一套向量引擎。当前向量能力的底层仍是 **DuckDB + VSS 扩展 + HNSW**；RSDuck 的作用是把这些底层能力组织成可供 Agent 线上使用的受管服务。

| 维度 | 纯 DuckDB 向量使用 | RSDuck 向量检索 |
|---|---|---|
| 使用方式 | 本地 SQL 或嵌入式程序调用 | HTTP Vector API 与受管 SQL |
| 向量字段 | 应用自行定义和校验 | `FLOAT[N]` 固定维度强校验 |
| 相似度查询 | 应用编写 `list_cosine_distance` 等 SQL | `search` API 返回 `memory_id + distance` |
| HNSW 索引 | 应用加载 VSS、创建索引并自行调参 | Catalog 记录定义、参数、状态和物理代次 |
| 索引异常 | 应用自行判断、告警和恢复 | 提供 `building`、`active`、`stale`、`failed`、`unavailable` 等明确状态 |
| 多租户边界 | SQL 中自行添加 `WHERE` | Token、租户、Agent、向量空间共同限制 |
| 重复消息处理 | 应用自行实现事件幂等 | `source_version + content_hash` 处理写入与删除事件 |
| 快照与恢复 | 应用自行恢复表、索引与版本关系 | 保存向量与索引定义，恢复后依据 Catalog 重建 HNSW |
| 灵活性 | 高，可直接使用 DuckDB 原生能力 | 受 Catalog 和 API 契约约束，换取一致的生产行为 |

前文的 SQL 就是典型的纯 DuckDB **精确检索**路径：对过滤范围内每一行计算距离，因此结果是精确的，但计算成本会随向量数近似线性增加。它适合调试、验收和作为 Recall 基准真值。

RSDuck 线上通常调用 `POST /api/vector/search` 并指定 `mode=ann`，使用 HNSW 进行近似最近邻检索。它以极小概率遗漏边缘候选为代价，换取大规模记忆下更快、更稳定的延迟。两条路径并不冲突：推荐用 RSDuck 的 ANN 做线上召回，用精确 SQL 做离线质量对比和故障排查。

选择建议很直接：单机分析、原型验证、数据量较小或需要完全自由 SQL 时，直接使用 DuckDB 更轻；多 Agent、多租户、Outbox 同步、长期运行并需要权限、索引状态、重建和恢复时，使用 RSDuck 更合适。

## 6. RSDuck 的亮点：不仅是“能建一个向量索引”

### 6.1 固定维度向量是可验证的数据契约

`FLOAT[N]` 在写入时校验维度，拒绝 NaN、Infinity；使用 cosine 时还拒绝零向量。它避免了不同模型、不同维度的向量被悄悄混入同一列，防止检索结果在数据层面失真。

### 6.2 HNSW 由 Catalog 管理，而非一次性 SQL 副作用

RSDuck 将向量空间、模型版本、距离算法、HNSW 参数、物理索引代次和运行状态记录在 `rsduck_catalog.rs_vector_index` 中。创建、重建、压缩和恢复都有明确状态：

```text
pending -> building -> active
                 |         |
                 v         v
              failed     rebuilding / compacting

VSS 版本变化：stale
物理索引缺失或中断恢复：unavailable
```

只有 `active` 索引可用于 ANN。索引构建中、VSS 不可用或索引失效时，服务返回稳定错误，而不会偷偷退化为全表精确扫描，把容量问题隐藏成偶发慢请求。

### 6.3 VSS 扩展受控加载

RSDuck 管理 DuckDB VSS 扩展，并限制只加载所需的 `vss`。扩展缺失、版本不匹配或加载失败会明确报错；发行服务包会携带对应平台的扩展文件。这样可以避免“开发机可用、用户机器启动后才发现索引不可用”的不确定性。

### 6.4 专用 Vector API 面向 Agent 事件流

Vector API 提供批量写入、删除、检索、索引状态、重建和压缩能力。写入以 `(tenant_id, agent_id, memory_id)` 为业务键，并以单调递增的 `source_version` 处理 Outbox 的至少一次投递：

- 重复事件可幂等成功；
- 旧事件不能覆盖新事件；
- 同版本、不同内容会明确冲突；
- 一批操作在同一事务中完成，失败不留下半批数据。

Token 还会按租户、Agent、向量空间及 `search/write` 权限限制，避免一个 Agent 误检索到另一个租户的记忆。

### 6.5 索引是派生数据，恢复路径清晰

Snapshot v3 保存向量数据、Catalog 和索引定义；恢复时首先恢复 `FLOAT[N]` 表，再依据 Catalog 重建物理 HNSW。这样快照不依赖某个不可移植的索引二进制文件，也能在平台或 VSS 版本变化后明确判断索引是否需要重建。

## 7. 适合哪些场景

| 场景 | RSDuck 在做什么 | 业务库仍负责什么 |
|---|---|---|
| 职业 Agent 长期记忆 | 根据当前问题召回相似历史记忆 | 原文、权限、业务状态与删除标记 |
| 客服或销售助手 | 找到相似工单、商机、客户偏好 | 客户主数据、工单流程、合规规则 |
| 知识库问答 | 召回相关文档片段 ID | 文档正文、版本、发布状态与 ACL |
| 研发助手 | 找到相关故障、决策记录和代码说明 | Issue、代码库、权限与最新状态 |
| 运营分析助手 | 找到类似活动、复盘和策略记录 | 指标事实、审批记录和最终报表 |

它尤其适合“先召回少量候选，再由业务规则过滤并生成回答”的系统。它不适合替代严格条件查询、账务一致性查询或需要绝对精确 Top-K 的主业务路径；这些仍应使用关系型事实库或显式 `exact` 检索。

## 8. 案例：职业 Agent 的客户记忆召回

假设一个职业 Agent 负责跟进企业客户。业务库已有记忆：

```text
memory_id=90001：客户要求将月度经营报表改为每周一上午发送。
memory_id=90002：客户对价格敏感，续约前需要准备套餐对比。
memory_id=90003：客户上周确认了数据接口白名单。
```

### 写入阶段

1. 业务服务先在 MySQL 写入记忆正文，并产生带版本号的 Outbox 事件。
2. Embedding Worker 读取事件，调用模型生成 `FLOAT[1536]` 向量。
3. Worker 调用 RSDuck `upsert-batch`，提交业务键、`source_version`、内容哈希和向量。
4. RSDuck 验证租户、Agent、维度和版本后，写入向量表，HNSW 负责后续 ANN 召回。

### 查询阶段

用户问：“那个客户的报告我们什么时候发？”

1. Agent 将问题生成同一模型的查询向量。
2. 调用 `POST /api/vector/search`，带上固定的 `tenant_id`、`agent_id`、`top_k` 和 `mode=ann`。
3. RSDuck 返回候选，例如 `90001 (distance=0.031)`、`90003 (distance=0.274)`。
4. Agent 回 MySQL 批量读取这些 ID，重新校验仍然可见且未删除，并按 RSDuck 的距离顺序重排。
5. Agent 使用 `90001` 的正文回答：“按已记录的约定，应在每周一上午发送。”

如果 `90001` 后续被业务侧修改或删除，新的 Outbox 事件会以更高 `source_version` 更新或删除 RSDuck 中的向量记录；旧事件即使延迟到达也不会把新数据覆盖回去。

## 9. 接入时最重要的五条规则

1. 只在同一个 `vector_space` 使用同一模型、同一版本、同一维度和同一距离算法。
2. 业务正文与权限始终放在事实源；RSDuck 只返回候选 ID 与距离。
3. 写入和删除必须携带单调递增的 `source_version`，超时重试使用原版本和原内容哈希。
4. 线上默认使用 `mode=ann`，索引非 `active` 时由调用方按明确策略重试或降级；RSDuck 不隐式全表扫描。
5. 模型升级创建新向量空间，全量回填、评估后切流，不能把不同模型的向量混写。

## 10. 结语

RSDuck 的价值不只是让 DuckDB “支持向量”。它把 Agent 记忆检索需要的固定维度约束、索引生命周期、VSS 依赖、租户隔离、事件幂等、快照恢复和性能基准放在同一套受管能力中。

对 Agent 系统而言，这意味着可以用一条清晰的路径把“业务事实”与“语义召回”分开：业务库保证真实与可控，RSDuck 让长期记忆在规模增长后仍然可快速找回。

具体的表结构、接口、认证配置、错误码和模型升级流程，请参阅 [Agent 向量记忆检索与索引接入规范](agent-vector-memory.md)。
